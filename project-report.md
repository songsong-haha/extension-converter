서버리스 클라이언트 파일 확장자 변환 앱 기획 리서치 보고서
Executive Summary
본 기획은 “사용자가 파일을 업로드하되, 파일 자체는 서버로 전송하지 않고(클라이언트에서만 처리) 확장자/포맷 변환을 제공하며, 광고로 수익화하는 서버리스(정적 호스팅 중심) 애플리케이션”을 목표로 한다. 핵심 가치는 (a) 민감 파일을 외부 서버에 업로드하지 않아도 되는 프라이버시 이점, (b) 설치 없이 즉시 사용 가능한 접근성, (c) 저비용 운영(백엔드 연산·저장 없이)이다. 유사 사례로 브라우저 내에서 로컬 처리(“이미지가 서버로 가지 않는다”)를 명시하는 Squoosh가 존재하며, 이 접근이 사용자 신뢰 형성에 기여할 수 있음을 시사한다.

기술적으로는 “브라우저의 File API/다운로드 + 점진적 향상(progressive enhancement)”을 기본 축으로 삼고, 고성능 변환은 WebAssembly(WASM) 기반 라이브러리(이미지: ImageMagick WASM, 비디오/오디오: FFmpeg WASM 등)를 워커에서 실행하는 구조가 합리적이다. 다만 멀티스레드 WASM(SharedArrayBuffer 기반)을 활성화하려면 cross-origin isolation(COOP/COEP)이 필요하고, 이는 광고·서드파티 리소스 로딩과 충돌할 수 있다(예: COEP는 CORP/CORS 미설정 크로스오리진 리소스를 차단). 따라서 초기 MVP는 (1) 멀티스레드에 의존하지 않는 변환 범위(이미지·일부 PDF 등) 중심으로 구성하고, (2) 광고는 “추적 최소화·비개인화(컨텍스추얼) 중심”으로 설계하되, (3) 지역 규제(특히 EU/UK 트래커 동의)와의 합치 여부를 초기부터 설계 변수로 두는 것이 위험을 낮춘다.

수익화 측면에서 ‘트래킹 없이’의 엄밀한 정의가 중요하다. 예컨대 Google의 “비개인화 광고(NPA)”는 과거행동 기반 개인화는 하지 않지만, 빈도 제한(frequency capping)·집계 리포팅·부정행위 방지 목적의 쿠키 사용이 있을 수 있으며, EU ePrivacy 적용 국가에서는 그 쿠키 사용에 동의가 필요하다고 명시한다. 즉 “무트래킹(쿠키/로컬ID 0)”을 목표로 한다면 광고 네트워크 선택·구현 방식이 근본적으로 달라질 수 있다.

이 보고서는 (1) 목표 사용자(페르소나)와 문제 정의, (2) 기능·UX 흐름, (3) 기술 아키텍처(플랫폼 제약 포함), (4) 광고 수익화와 프라이버시/법적 고려, (5) 보안·프라이버시·법적 리스크 대응, (6) MVP/로드맵·경쟁 분석·비용/수익 모델·테스트·확장성을 통합한 실행 계획을 제시한다. “미정” 항목은 명시적으로 “미정”으로 표기한다.

제품 정의

파일 I/O: 점진적 향상과 폴백
기본 경로는 <input type="file"> 기반 로컬 읽기 + 변환 결과를 Blob으로 생성 후 다운로드 링크로 제공하는 방식이다.
가능하면 File System Access API를 활용해 “저장(Save As)”에 가까운 UX를 제공할 수 있다. 이 API는 로컬 파일/디렉터리 접근 핸들을 통해 읽기·쓰기·저장을 지원한다.
그러나 브라우저 지원이 제한적이므로, “지원 시 사용 + 미지원 시 legacy 방식 폴백”이 필요하다. 이를 위해 browser-fs-access는 File System Access API를 사용하되 미지원 브라우저에서는 <input>과 <a download>로 폴백한다는 설계를 명시한다.
Chromium 계열(데스크톱/안드로이드)에서 지원이 강하고, 다른 브라우저는 부분 지원/제약이 존재할 수 있음을 전제로 한다.
변환 엔진 선택: 브라우저 네이티브 vs WASM
이미지 변환의 1차 MVP는 브라우저 네이티브 Canvas 기반이 비용/안정성 측면에서 유리하다. HTMLCanvasElement.toBlob()은 캔버스 내용을 Blob으로 생성하는 표준 메서드다.

다만 포맷별 인코더 지원은 브라우저마다 다르며, 예컨대 toBlob()의 type 파라미터로 WebP를 지정하는 기능은 브라우저 지원 편차(특히 Safari)가 크다고 보고된다.

고급 포맷(다양한 이미지/문서/멀티미디어)을 폭넓게 지원하려면 WASM 기반 라이브러리가 현실적이다.

이미지: magick-wasm은 ImageMagick을 웹에서 “API 콜백 없이” 사용할 수 있다고 명시한다.
PDF: pdf-lib은 TypeScript로 작성되어 순수 JS로 컴파일되며 브라우저 등 다양한 런타임에서 동작한다고 안내한다.
오디오/비디오: ffmpeg.wasm은 FFmpeg를 WebAssembly/JS로 포팅하여 브라우저 내에서 녹화/변환/스트리밍을 가능하게 한다고 설명한다.
성능/호환성: 워커, 메모리 한계, iOS/Safari 리스크
WASM 변환은 메인 스레드를 장시간 점유할 수 있으므로 Web Worker에서 실행하는 것이 바람직하다(UX/응답성). 멀티스레드 WASM(Threads)을 쓰면 속도가 개선될 수 있으나, SharedArrayBuffer가 필요하며 이는 “secure context + cross-origin isolated” 요건을 가진다.

여기서 중요한 설계 충돌이 발생한다.

COEP는 문서가 크로스오리진 리소스를 로드하려면 CORP 또는 CORS 헤더가 있어야 한다는 정책을 강제할 수 있으며, iframes/nested frames에도 적용된다.
web.dev의 cross-origin isolation 가이드는, cross-origin isolation이 사이트의 다른 크로스오리진 리소스(예: 광고 배치)에 미치는 영향을 평가하라고 직접 경고한다.
따라서 “고성능(멀티스레드) + 서드파티 광고(대부분 크로스오리진) + 단일 페이지”를 동시에 만족시키기 어렵다. 이를 반영해 초기에는 멀티스레드 의존도를 낮추고(또는 기능 플래그로 분리) 광고와의 충돌을 최소화하는 방향이 위험을 줄인다.

또한 iOS/Safari 계열 제약(메모리·스토리지·WASM threads 등)은 파일 변환 제품에서 치명적일 수 있다.

WebAssembly는 wasm32 기준 최대 4GB 메모리 한계를 설명하며, memory64는 진행 중이지만 Safari에서의 지원이 제한될 수 있다.
WebKit은 저장 공간 정책/eviction이 발생할 수 있음을 설명하고(스토리지 압박 등 조건에서 자동 삭제), Safari 17+의 quota/eviction 정책을 공개한다.

광고 수익화 모델과 프라이버시 설계

요구사항 충족을 위해 다음 두 수준을 분리해 정의하는 것이 실무적으로 중요하다.

Level 1: “파일 콘텐츠는 서버로 전송하지 않음” (본 제품의 핵심 약속)
Level 2: “행태정보/식별자 기반 추적을 하지 않음(또는 최소화)” (광고·분석 설계의 핵심)
한국 맥락에서 온라인 맞춤형 광고 가이드라인은 행태정보 처리의 투명성·통제권 보장·안전성 확보 등을 강조하며, 행태정보를 개인 식별정보와 결합해 사용할 경우 사전 동의를 요구한다.

또한 「개인정보 보호법」은 개인정보의 정의(다른 정보와 쉽게 결합해 특정 개인을 알아볼 수 있는 정보 포함)를 규정한다.

즉 광고 스택이 쿠키/광고식별자 등으로 사용자를 식별·추적하거나 이를 다른 정보와 결합하는 순간, 법적·프라이버시 리스크가 크게 상승한다. Level 2를 달성하려면 “광고 방식 자체를 컨텍스추얼 중심으로 제한”하거나 “직접 판매형 광고”로 초기화하는 접근이 필요하다.

광고 격리와 안전한 렌더링
광고는 “파일을 다루는 실행 컨텍스트”와 가능한 한 분리해야 한다. 실무적 이유는 (a) 광고 스크립트/서드파티 코드가 DOM 접근을 통해 민감 데이터(파일 메타데이터/사용자 입력)를 노출시킬 가능성, (b) 악성 광고가 리디렉션/팝업/다운로드를 유발할 가능성 때문이다. SafeFrame은 퍼블리셔 페이지와 광고 크리에이티브를 분리하면서 API로 제한된 상호작용만 허용하는 방식으로 설명되며, 샌드박싱을 통해 자동 리디렉션·팝업·자동 다운로드 등 원치 않는 행위를 줄일 수 있다고 기술한다.

다만 SafeFrame/iframe 격리 역시 COEP 등 cross-origin isolation 설계와 충돌할 수 있으므로, 초기에는 다음 절충이 현실적이다.

P0: “광고가 필요한 화면”과 “파일 변환 실행(Worker/WASM) 화면”을 논리적으로 분리하고, 변환 화면에서는 광고 로딩을 최소화(또는 비활성)한다. cross-origin isolation 가이드가 광고 영향 평가를 요구하는 점과 정합적이다.
P1: 광고는 sandboxed iframe으로 렌더링하되, 허용 권한을 최소화한다(예: top-navigation은 user activation일 때만 허용). SafeFrame 문서에서도 allow-top-navigation-by-user-activation 등 권장 속성을 제시한다.
지역별(법/정책) 고려의 핵심 포인트
한국: 「개인정보 보호법」의 개인정보 정의는 결합 가능성까지 포함하므로, 쿠키/광고식별자/행태정보가 다른 정보와 결합될 가능성을 낮추는 설계가 요구된다.
맞춤형 광고·행태정보: 방송통신위원회 보도자료(현재는 소관이 개인정보보호위원회로 변경됨)는 행태정보 수집·이용의 투명성, 이용자 통제권, 안전성 확보 원칙 등을 제시한다.
EU/UK(글로벌 서비스 시): Google은 EU user consent policy에서 NPA라도 빈도제한/집계 리포팅/부정행위 방지 목적 쿠키 사용이 있을 수 있고, ePrivacy 적용 국가에서 그 쿠키 사용에 동의가 필요하다고 명시한다.
따라서 “국내 전용 서비스”인지 “글로벌 공개”인지가 수익화 설계의 선결 변수다. 현재는 미정으로 두되, MVP 단계에서 반드시 확정해야 한다(미정).

리스크와 대응 방안: 보안·프라이버시·법적
핵심 리스크 식별
본 제품은 “서버로 파일을 보내지 않는다”는 약속을 중심으로 신뢰를 구축하므로, 신뢰 붕괴(실제 유출이 없더라도 오해/논란 포함)가 가장 큰 비즈니스 리스크다. 특히 광고/서드파티 스크립트가 존재하는 순간, 사용자는 “결국 외부로 뭔가 나간다”는 인상을 받을 수 있다.

보안적으로는 (1) XSS/서드파티 스크립트 공급망, (2) 악성 광고/리디렉션, (3) 변환 라이브러리(WASM) 취약점, (4) 로컬 저장소(IndexedDB) 잔존 데이터, (5) 브라우저 권한(File System Access) 오남용이 주요 축이다. CSP는 XSS 등 공격에 대한 방어 심층화(Defense in Depth) 수단으로 강조된다.

또한 서드파티 리소스 무결성 측면에서 SRI(Subresource Integrity)는 CDN 등 외부 리소스가 변조되지 않았음을 해시로 검증한다.

테스트 계획
기능 테스트: 포맷별 정상 변환/메타데이터 유지·삭제/다운로드 무결성(파일 해시 비교, 미정)
호환성 테스트: Chromium/Firefox/Safari, iOS/Android 실제 기기 중심(특히 Safari 인코딩/저장소)
성능 테스트: 파일 크기/해상도별 처리시간, 워커 사용 여부, 메모리 OOM 임계점 탐색
보안 테스트: CSP 적용 검증, 서드파티 스크립트/SRI 검증, 클릭재킹/리디렉션 방지(광고 iframe sandbox)
측정 지표(서버리스 제약 반영)
서버가 없으면 “자체 로그”를 모으기 어렵다. 선택지는 다음 두 갈래이며, 현재는 미정이다.

선택지 A(무서버 엄수): 제품 내 계측 최소화, 광고 플랫폼 리포트(노출/클릭)만으로 간접 성과 추정(정밀도 낮음, 미정).
선택지 B(서버리스/외부 서비스 활용): 프라이버시 친화 분석 도구를 도입하되, 쿠키·식별자를 최소화하고 동의/거부를 제공(법적 검토 필요). EU에서는 분석 쿠키도 요건이 까다로울 수 있으며, CNIL은 조건부로 ‘동의 면제(옵트아웃)’가 가능한 사례를 안내하지만 엄격한 조건을 부과한다.
MVP에서는 선택지 A로 시작하고, 유입이 확인되면 선택지 B를 검토하는 단계적 접근이 리스크를 낮춘다(미정).

확장 가능성
오프라인 기능 확장: 서비스 워커로 정적 자산과 WASM 번들을 캐시하여 “한 번 로드 후 오프라인 변환”을 제공할 수 있다. 서비스 워커는 오프라인 동작을 위한 리소스 캐싱을 가능하게 한다.
플러그인/SDK: 변환 엔진을 “포맷 플러그인” 인터페이스로 추상화하면, 신규 포맷 추가를 독립적으로 배포할 수 있다(예: image/\* 코덱, pdf 도구, audio/video). 다만 브라우저 저장소/캐시 정책(특히 Safari)과 번들 용량의 상충을 고려해야 한다.
오픈소스 가능성: 신뢰 확보에 유리하다. Squoosh는 오픈소스이며 “이미지는 서버로 보내지 않는다”는 프라이버시 섹션을 명시한다.

본 기획의 초점은 “로컬 처리 신뢰”를 훼손하지 않으면서도 광고 수익화를 가능하게 하는 균형점에 있다. 기술적으로는 점진적 향상(파일 시스템 API 폴백), WASM 도입의 단계화, cross-origin isolation과 광고의 충돌 관리가 핵심이며, 프라이버시·법적 관점에서는 행태정보/식별자 최소화와 투명성·통제권 제공이 필수 전제다.
